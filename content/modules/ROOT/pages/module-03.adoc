## Step 3: Understanding Audit Logs

Audit logs contain detailed records of security-relevant events on your system. Understanding how to read and interpret these logs is essential for security monitoring and compliance.


**3.1** Audit logs are stored in `/var/log/audit/`. Let's examine this directory:
[source,bash,role="execute",subs=attributes+]
----
sudo ls -lahn /var/log/audit/
----

The primary log file is `audit.log`. Depending on your configuration, you may see rotated logs like `audit.log.1`, `audit.log.2`, etc.


**3.2** Let's look at recent audit log entries:
[source,bash,role="execute",subs=attributes+]
----
sudo tail -20 /var/log/audit/audit.log
----

Audit log entries can seem cryptic at first, but they follow a consistent structure. Each entry contains:

* **Timestamp** - When the event occurred
* **Record Type** - What kind of event (SYSCALL, PATH, USER_AUTH, etc.)
* **Event-specific fields** - Details about the event

[TIP]
====
Don't be intimidated by the raw log format! The `ausearch` and `aureport` tools (which we'll use later) make it much easier to find and interpret audit events.
====

Audit logs use different record types to categorize events. Here are the most important ones:

=== SYSCALL Records

SYSCALL records capture system call events. They include:

* **uid** - User ID who made the call
* **tty** - Terminal if interactive
* **success** - Whether the call succeeded (yes/no)
* **exe** - The executable that made the call
* **comm** - Command name

Example:
[source,text]
----
type=SYSCALL msg=audit(1234567890.123:456): arch=c000003e syscall=2 success=yes exit=3
  a0=7ffe1234abcd a1=0 a2=0 a3=7ffe1234def0 items=1 ppid=1234 pid=5678
  auid=1000 uid=0 gid=0 euid=0 suid=0 fsuid=0 egid=0 sgid=0 fsgid=0
  tty=pts0 ses=1 comm="cat" exe="/usr/bin/cat" key=(null)
----

=== CWD Records

**3.3** CWD (Current Working Directory) records show where a process was running when it triggered an audit event:
[source,bash,role="execute",subs=attributes+]
----
type=CWD msg=audit(1234567890.123:456): cwd="/home/rhel"
----

=== PATH Records

**3.4** PATH records capture file paths that were accessed or passed as arguments to system calls:
[source,bash,role="execute",subs=attributes+]
----
type=PATH msg=audit(1234567890.123:456): item=0 name="/etc/passwd"
  inode=12345 dev=fd:00 mode=0100644 ouid=0 ogid=0 rdev=00:00
  nametype=NORMAL
----


**3.5** PROCTITLE records show the full command-line that triggered the audit event:
[source,text]
----
type=PROCTITLE msg=audit(1234567890.123:456): proctitle=2F7573722F62696E2F636174002F6574632F706173737764
----

[NOTE]
====
PROCTITLE values are hex-encoded. The `ausearch` tool automatically decodes them for readability.
====


**3.6** Let's use `ausearch` to find specific events. For example, search for authentication events:
[source,bash,role="execute",subs=attributes+]
----
sudo ausearch -m USER_AUTH -ts today | head -30
----

This searches for:
* `-m USER_AUTH` - Authentication record type
* `-ts today` - Events from today
* `head -30` - Shows first 30 lines

[TIP]
====
The `ausearch` tool supports many search criteria:

* `-m` - Message type (USER_AUTH, SYSCALL, etc.)
* `-ts` - Start time (today, recent, yesterday, or a specific timestamp)
* `-te` - End time
* `-k` - Search by key (we'll create keyed rules later)
* `-f` - Search by filename
* `-ui` - Search by user ID
====
