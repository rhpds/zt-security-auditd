= Custom Rule Creation

== Overview

While preconfigured rules are excellent for compliance, you'll often need to create custom audit rules tailored to your organization's specific security requirements. In this module, you'll learn how to create persistent, custom audit rules.

== Understanding Rule File Naming

Rule files in `/etc/audit/rules.d/` follow a priority-based naming convention:

[source,text]
----
<priority>-<name>.rules
----

Priority ranges and their purposes:

* **10** - Kernel and buffer configuration
* **20** - Rules that must match before more general rules
* **30** - Main rules
* **40** - Optional rules
* **50** - Server-specific rules
* **70** - Local/custom rules
* **90** - Finalize rules

[TIP]
====
Use priority **70** for your custom local rules. This ensures they load after the main STIG rules but before finalization.
====

== Step 1: Create a Temporary Rule

First, let's create a rule to monitor the `ping` command and see it in action:

[source,bash,role=execute]
----
sudo auditctl -a always,exit -F exe=/usr/bin/ping -k rhkey
----

This rule:
* **`-a always,exit`** - Adds a rule that always logs at syscall exit
* **`-F exe=/usr/bin/ping`** - Filters for the ping executable
* **`-k rhkey`** - Tags events with the key "rhkey"

== Step 2: Test the Rule

Let's trigger the rule by pinging localhost:

[source,bash,role=execute]
----
ping -c 2 localhost
----

Now search for the audit events:

[source,bash,role=execute]
----
sudo ausearch -k rhkey -ts recent
----

You should see audit records showing the ping command execution!

== Step 3: Make the Rule Persistent

The rule we just created is temporary. Let's make it persistent by creating a rule file:

[source,bash,role=execute]
----
sudo touch /etc/audit/rules.d/70-rhkey_lab.rules
----

Add the rule to the file:

[source,bash,role=execute]
----
echo "-a always,exit -S all -F exe=/usr/bin/ping -F key=rhkey" | sudo tee /etc/audit/rules.d/70-rhkey_lab.rules
----

[NOTE]
====
Notice the slight syntax difference in the file:
* Command line: `-k rhkey`
* Rule file: `-F key=rhkey`

Both achieve the same result, but rule files use the `-F` format for consistency.
====

== Step 4: Reload Rules

Load the new rule file:

[source,bash,role=execute]
----
sudo augenrules --load
----

Verify the rule is loaded:

[source,bash,role=execute]
----
sudo auditctl -l | grep rhkey
----

== Step 5: Adjust Kernel Buffer Size

For systems with high audit activity, you may need to increase the kernel's audit backlog buffer to prevent event loss.

First, set the buffer size temporarily:

[source,bash,role=execute]
----
sudo auditctl -b 8192
----

To make this persistent, create a kernel configuration rule file:

[source,bash,role=execute]
----
sudo touch /etc/audit/rules.d/15-rhkey_kernel.rules
----

Add the buffer configuration:

[source,bash,role=execute]
----
echo "-b 8192" | sudo tee /etc/audit/rules.d/15-rhkey_kernel.rules
----

[TIP]
====
The `-b` option sets the maximum number of outstanding audit buffers. The default is often too small for busy systems. Increase it if you see "backlog limit exceeded" messages.
====

== Step 6: Reload and Verify

Reload all rules to include the kernel configuration:

[source,bash,role=execute]
----
sudo augenrules --load
----

Verify the current backlog limit:

[source,bash,role=execute]
----
sudo auditctl -s | grep backlog
----

== Common Rule Patterns

Here are some useful rule patterns you can adapt:

**Monitor file access:**
[source,text]
----
-w /etc/passwd -p wa -k passwd_changes
----

**Monitor directory:**
[source,text]
----
-w /etc/ssh/ -p wa -k ssh_config
----

**Monitor syscall by executable:**
[source,text]
----
-a always,exit -F arch=b64 -S execve -F exe=/usr/bin/sudo -k sudo_exec
----

**Monitor by user ID:**
[source,text]
----
-a always,exit -F arch=b64 -S all -F uid=1001 -k user_1001_activity
----
